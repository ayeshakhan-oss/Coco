## Objective

Summarize hiring recommendations and candidate verdicts after all interview rounds (CV screening → case study → values → debrief) are complete. Decision Brief goes to hiring manager and stakeholders to inform final hiring decisions.

**Format:** Inline HTML email (no PDF attachment)

**Timing:** When all interview rounds are done and verdicts are finalized

---

## Decision Brief Sections (4-Part Structure)

### Part A: Header & Stat Boxes

**Header Block:**
- Dark navy background (#1a2a3a or similar)
- Small uppercase gray text: "People & Culture · Decision View"
- Large white title: [Job Title]
- Light blue subtitle: "Job [X] · Taleemabad"

**Stat Boxes (4 colored boxes, horizontal layout):**
1. **Total candidates screened** (red/pink background) — count all received
2. **Total values calls conducted** (blue background) — count all values interviews done
3. **Total shortlisted** (yellow/orange background) — count final shortlist
4. **Decision status** (gray background) — e.g., "5 debrief calls completed" OR "Panel decision pending"

---

### Part B: Leading Candidates (Top Recommendations)

**Purpose:** Present the strongest candidates with clear rationale for hiring manager to prioritize.

**Structure per candidate:**

1. **Name (hyperlinked to Google Drive CV)** — use cv_link(candidate_name) helper
   - Current Role & Company
   - Key Strengths (2–3 bullet points tied to JD requirements)
   - Specific example: "Demonstrates X directly relevant to our Y; case study showed Z"

2. **Case Study Performance** — 1–2 sentences
   - Format: "Tier 1 on case study (87%). Strong synthesis and methodological thinking across all exercises."

3. **Values Assessment** — PASS or CONDITIONAL GWC
   - Format: "Values PASS. GWC: YES. Ready for Right Seat."
   - Or: "Values PASS. GWC: CONDITIONAL on alignment check regarding remote work expectations."

4. **Debrief Verdict** — what the debrief call revealed
   - Verdict labels (use EXACTLY): "PANEL DECISION" / "DEBRIEF CONFIRMED" / "DEBRIEF SCHEDULED" / "OVERDUE"
   - 1–2 observations from debrief if completed

5. **Recommendation** — clear next action
   - Example: "Recommend for offer. Ready to move forward pending final approvals."
   - Or: "Hold for Right Seat conversation. Strong candidate, minor alignment gaps to clarify."

---

### Part C: Pipeline Summary (All Other Candidates)

**Purpose:** Show what happened to every other candidate. Grouped by status.

**Group 1: VALUES PASS but Debrief Pending**
- List candidates by name (hyperlinked)
- Case study score in parentheses
- Debrief date scheduled or "Awaiting confirmation"
- Example: "Muhammad Hassan (Tier 1, 84%) — Debrief scheduled 2026-04-15"

**Group 2: VALUES OUT**
- List candidates by name (hyperlinked)
- Reason: specific value gap (e.g., "Lacks Courageous Conversations evidence")
- Case study tier for context
- Example: "Amara Khan (Tier 1 case study, 81%) — VALUES OUT: Inconsistent on All for One (multiple +/- ratings)"

**Group 3: CASE STUDY OUT / Did Not Advance**
- List candidates by name (hyperlinked)
- Case study tier and reason
- Example: "Bilal Ahmed (Tier 3, 62%) — Case study performance below threshold. Values interview not conducted."

**Group 4: OVERDUE / Pending Submission**
- Candidates who haven't submitted case study yet or missed deadlines
- Example: "Sarah Patel — CASE STUDY OVERDUE since 2026-04-05. Sent reminder 2026-04-09. No response."

**Group 5: NOT INTERVIEWED**
- Candidates screened but didn't advance to values interview
- Example: "Ahmed Malik — CV screened, not shortlisted (Tier 3, 58% JD match). Did not advance to values interview."

---

### Part D: Debrief Schedule & Next Steps

**If debrief calls still pending:**
- List candidates with debrief dates scheduled
- Include Teams link if available
- Format: "Name — [Date] [Time] via Teams"
- Example: "Rosheen Naeem — 2026-04-11 10:00am via Teams"

**If all debrief calls complete:**
- State: "All scheduled debrief calls completed as of [date]."
- Next step: "Awaiting final panel decision."
- Or: "Final panel decision required by [date]."

**Final action items:**
- "Hiring manager to confirm final offer candidate by [date]"
- "All candidates to be notified by [date]"
- "Onboarding timeline: [date]"

## Non-Negotiable Rules

1. **Every candidate name must be hyperlinked** — In every section (Leading, Pipeline, Debrief Schedule), every name must link to Google Drive CV. Audit before sending.

2. **Verdict labels must match approved list EXACTLY:**
   - PASS = "VALUES PASS"
   - OUT = "VALUES OUT"
   - Debrief status = "DEBRIEF TODAY" / "DEBRIEF CONFIRMED" / "DEBRIEF SCHEDULED" / "OVERDUE"
   - Final decision = "PANEL DECISION"
   - Case study in = "CASE STUDY IN"
   - Case study sent = "CASE STUDY SENT"
   - Not interviewed = "NOT INTERVIEWED"
   - **NEVER use:** "OFFER OUT", "OFFER STAGE", or any undefined label

3. **DB status ≠ actual status** — If Markaz shows status='offer', do NOT assert that an offer was sent. Always verify with Ayesha. Same for status='rejected' — flag as potential data entry error. Never assert DB status as truth in the brief.

4. **GWC must be explicit** — If values PASS, state: "GWC: YES" or "GWC: CONDITIONAL on [specific reason]". Never leave GWC blank.

5. **Case study scores are context only** — Include tier and percentage for ranking, but do not re-explain case study logic. That was in the case study report.

6. **All observations must be factual** — Cite specific case study evidence, interview moments, debrief findings. No assumptions or "probably" language.

7. **Email format is HTML inline** — Use v8 design (blue headings, Georgia serif, stat boxes). No PDF attachment.

8. **Email recipients:** TO = hiring manager, CC = hiring@taleemabad.com + ayesha.khan@taleemabad.com (+ any additional stakeholders user specifies)

9. **If thread reply:** Use In-Reply-To and References headers to stay in original thread. Subject = "Re: [original]".

10. **Always ask for approval** — PILOT mode first (send to Ayesha + Jawwad only). User reviews and approves. Then LIVE send. No exceptions.

## Common Mistakes

1. **Candidate names not hyperlinked** — "Ahmed Hassan" instead of clickable CV link. Hiring manager needs one-click access.

2. **Using wrong verdict labels** — "OFFER OUT" instead of "PANEL DECISION" or "DEBRIEF CONFIRMED".

3. **Asserting DB status as final truth** — "Status in Markaz shows 'offer', so they have an offer." Never. Always verify.

4. **GWC blank or unclear** — "Values PASS" without stating if GWC is YES or CONDITIONAL.

5. **Mixing case study narrative** — Case study report detailed exercises. Brief is summary. Don't re-explain.

6. **Candidate groups unclear** — Pipeline section confusing because statuses aren't organized. Use clear headers.

7. **Next steps vague** — "Awaiting decision" too soft. State exact date and who decides.

8. **PILOT_MODE = False immediately** — sent straight to hiring manager without Ayesha review.

9. **Missing CV links for out candidates** — Only leading need links? Wrong. Every name needs a link.

10. **Sending without approval** — PILOT first, approval, then live.
